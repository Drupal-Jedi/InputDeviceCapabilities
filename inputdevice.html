The APIs introduced by this document provide authors with a way to determine the properties of different input devices in response to DOM input events.

This proposal was produced in collaboration with members of the W3C Web Applications Working Group, the Pointer Events Working Group and the Touch Events Community Group. It is my hope that the work here will ultimately be integrated into the [[UIEvents]] specification.

Although there is a wide variety of properties which would be valuable to add here, for now this document aims to define only those that are strongly justified as solving important real-world problems.

Introduction

DOM input events are an abstraction above low-level input events, only loosely tied to physical device input (e.g. ‘click’ events can be fired by a mouse, touchscreen or keyboard). There is no mechanism to obtain lower-level details about the physical device responsible for an event. This leads to problems, as developers may need to make assumptions or rely on heuristics. For example, when supporting both mouse and touch input, it's difficult to know whether a mousedown event represents new input from a mouse, or a compatibility event for a previously processed touchstart event.

The InputDevice API is designed to provide additional details about the underlying source of an input event. To avoid reliance on faulty assumptions (which may be violated by edge cases or future input devices), this API attempts to describe how a device behaves (e.g. whether it fires touch events), not what it is (e.g. a touchscreen).

The InputDevice interface

An InputDevice represents an input device (or group of related devices) capable of generating input events. Events caused by the same physical input device will get the same sourceDevice object, but the converse isn't necessarily true. E.g. two mice with the same properties in a system may appear as a single InputDevice instance.

In some cases an InputDevice will represent a logical device rather than a physical input device. For example, in some environments keyboard events are generated by tapping on an on-screen keyboard on a touchscreen. In such a scenario the InputDevice represents the properties of the virtual keyboard (since that's really what matters to authors), not the properties of the physical touchscreen device (which should be considered an implementation detail of the logical keyboard device). 

        [Constructor(optional InputDeviceInit deviceInitDict)]
        interface InputDevice {
          readonly attribute boolean firesTouchEvents;
        };

        dictionary InputDeviceInit {
          boolean firesTouchEvents = false;
        };

InputDevice attributes

firesTouchEvents
Indicates whether this device dispatches [[Touch-Events]].
This doesn't necessarily mean the device is a touch screen. For example, stylus and mouse devices typically generate touch events on mobile browsers. This property can be used to detect mouse events [[UIEvents]] which represent an action that may have already been handled by touch event handlers. See Example 1 for details.

Extensions to the UIEvent interface and UIEventInit dictionary

UIEvent and UIEventInit are defined in [[!UIEvents]].

        partial interface UIEvent {
          readonly attribute InputDevice? sourceDevice;
        };

        partial dictionary UIEventInit {
          InputDevice? sourceDevice = null;
        };
sourceDevice
Indicates the InputDevice responsible for the generation of this event, or null if no input device was responsible.
The exact semantics of when a device is "resposible" for an event depends somewhat on host OS properties and so is implementation-defined. For example, although a mouse or keyboard may be used to resize a window, this detail is not exposed to the web platform in any way and so the sourceDevice of a resize event will typically be null.

Examples

The following are examples that demonstrate how the APIs in this specification might be used

myButton.addEventListener('touchstart', function(e) {
  myButton.classList.add("pressed");
});

myButton.addEventListener('touchend', function(e) {
  // We don't call preventDefault because we want to rely on the browser's
  // tap detection to actually trigger the button.
  myButton.classList.remove("pressed");
});

myButton.addEventListener('touchcancel',  {
  myButton.classList.remove("pressed");
});

myButton.addEventListener('mousedown', function(e) {
  // Touch event case handled above, don't change the style again on tap.
  if (!e.sourceDevice.firesTouchEvents)
    myButton.classList.add("pressed");
});

document.addEventListener('mouseup', function(e) {
  if (!e.sourceDevice.firesTouchEvents)
    myButton.classList.remove("pressed");
});

Relationship to APIs in other platforms

Windows 8

Windows Runtime has a PointerDevice class, which is correlated to a pointer input event via Windows.UI.Input.PointerPoint.PointerDevice. Additional information is available from the MouseCapabilities, KeyboardCapabilities and TouchCapabilities classes. Note that this information is not available per-device.

At the low level, all WM_POINTER event messages provide access to a sourceDevice HANDLE for each point. Detailed per-device information can be obtained using the Raw Input APIs.

Mac OS X

MacOS X has various properties directly on NSEvent for determining device details. E.g. the mouse event sub-type, hasPreciseScrollingDeltas, and various properties (such as vendorPointingDeviceType) for tablet device information.

Android

Android has a rich InputDevice API similar to what’s being proposed here. Applications can determine the InputDevice for an event via InputEvent.getDevice, and can watch for changes in input devices using an InputDeviceManager.
